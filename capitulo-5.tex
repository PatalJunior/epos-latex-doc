\chapter{Use Case: EPOS API}
\label{chap:caso-epos}

This chapter applies the framework of Chapter~\ref{chap:metodo-arq} to the
production \ac{API}, \texttt{GLASS V4}. The \ac{EPOS} itself~\cite{epos} is a
pan-European research infrastructure in solid Earth science, organised into
Thematic Core Services for seismology, geomagnetic observations, \ac{GNSS} data
and products~\cite{eposGnss}, and so on. The \ac{SEGAL} laboratory of \ac{UBI}
contributes to the \ac{GNSS} service by operating a node that hosts station
metadata and \ac{RINEX}~\cite{rinex} observation files. \texttt{GLASS V4} serves
as the public interface of that node.

Three properties of this domain shape the testing framework: the entities
are relatively static (making stable baselines easy to obtain), typical client
requests return extensive lists of records (requiring the comparison
algorithm to scale to large diffs), and the same information is exposed
across multiple serialization formats (multiplying the testing surface area without
changing the underlying logic).

\section{Architecture of \texttt{GLASS V4}}
\label{sec:apiv4-arch}

The application is a stateless Jakarta EE 11 \ac{REST} service targeting
Java SE 21. It is packaged as a Maven \ac{WAR} (\texttt{target/ROOT.war})
and deployed onto Payara Server 6, anchoring the \ac{JAX-RS} application at
the root context via \texttt{@ApplicationPath("/api")}.

Its source tree is organised into a strict layered request pipeline where no
state is held between calls. This stateless design is critical, as it allows
the differential testing harness to boot parallel application instances simultaneously.
As shown in Figure~\ref{fig:apiv4-layers}, the architecture implements a "validate-then-query"
pattern, ensuring malformed requests are rejected with a \texttt{400 Bad
Request} before initiating costly database round-trips. Furthermore, the core
is format-agnostic: queries return raw entities, and the requested format (\ac{JSON},
\ac{XML}, \ac{CSV}, etc.) is applied only at final serialization.

\subsection{Endpoints and Output Formats}

The \ac{REST} surface consists of four primary resource families. \texttt{Start.java}
registers a \texttt{PingResource} for liveness checks (executing a trivial \texttt{SELECT
1} against the database). The core domain is handled by \texttt{StationEndpoints}
(\texttt{/api/stations}) and \texttt{FilesEndpoints} (\texttt{/api/files}). Finally,
\texttt{ListEndpoints} (\texttt{/api/list}) provides controlled
vocabularies and lookups exclusively in \ac{JSON}.

Endpoints follow a uniform grammatical structure driven by path parameters (e.g.,
\texttt{/stations/<selector>/.../\{size\}/\{format\}}). Selectors include identifiers,
locations, agencies, and coordinates. The \texttt{combination} endpoints
offer the highest flexibility, allowing an open set of query parameters validated
against a strict allow-list. For file endpoints, responses are paginated,
carrying metadata headers such as \texttt{X-Total-Count}, \texttt{X-Page},
and \texttt{X-Total-Pages}.

A single switch statement in each resource resolves the output format matrix.
Stations can be serialized to \ac{JSON}, \ac{XML}, \ac{CSV}, GeoJSON, or JSON-LD.
Files support \ac{JSON}, \ac{XML}, \ac{CSV}, and executable shell scripts. To
prevent structural drift, formats like \ac{JSON}, GeoJSON, and JSON-LD share
the same serialization logic, manipulated via boolean flags. Empty result sets
actively short-circuit to a \texttt{200 OK} plain-text message rather than
an empty document.

\subsection{Persistence}

Persistence is managed by Hibernate over PostgreSQL, utilising a \texttt{RESOURCE\_LOCAL}
unit named \texttt{MeuPU}. The connection configuration is resolved dynamically
at startup by \texttt{Config.Loader}, following a strict precedence: an
explicit \texttt{DATABASE\_URL} environment variable overrides individual host/port
variables, which override local configuration files, which finally fall back
to hard-coded defaults (\texttt{192.168.168.30:5432}).

This environment-variable precedence is the linchpin of the differential
harness: it enables the pipeline to point multiple historical builds at a single,
shared database purely by injecting \texttt{DATABASE\_URL} at runtime,
requiring no artifact rebuilds. As a safeguard, the pipeline enforces
\texttt{JAKARTA\_SCHEMA\_ACTION=none} to ensure Hibernate never executes \ac{DDL}
statements against live data.

To mitigate performance bottlenecks across complex joins, the architecture heavily
employs batch fetching (\texttt{hibernate.default\_batch\_fetch\_size = 512}).
This collapses potentially hundreds of thousands of lazy-loading \texttt{N+1}
queries into highly efficient \texttt{IN (...)} clauses.

\section{The CI/CD Pipeline Flow}
\label{sec:pipeline}

The continuous integration and deployment lifecycle is orchestrated by GitLab,
but the actual compilation and testing workloads are offloaded to a self-managed
internal runner (using the \texttt{ubi-runner} image). This runner is
provisioned within the laboratory's infrastructure, granting it secure \ac{VPN}
access to the reference PostgreSQL databases.

Before detailing the specific pipeline stages, it is important to understand
the execution flow between the control plane and this runner, as
illustrated in Figure~\ref{fig:gitlab-runner}. The runner process operates in
a continuous polling loop. Once registered, it periodically requests jobs from
the GitLab \ac{API}. When a developer pushes a new commit, GitLab queues
the pipeline jobs; the runner claims them, pulls the job payload, and delegates
the workload to its local executor. The executor then clones the repository,
downloads any necessary artifacts from previous stages, runs the defined
scripts, and streams the status and output back to GitLab.

\end{figure}

Mapped onto this execution architecture, our specific pipeline is divided into
three sequential stages: \texttt{build}, \texttt{test}, and \texttt{diff}.

\subsection{Stage 1 \& 2: Build and Unit Testing}
The \texttt{build} stage initiates the process. The executor compiles the candidate
\texttt{HEAD} \ac{WAR} and publishes it back to GitLab as a pipeline artifact.
Following this, the \texttt{test} stage downloads that artifact and
executes standard JUnit 5 logic tests. Crucially, these tests are entirely
isolated from the database. A deliberate design choice was made to forgo traditional
data-access integration tests; instead, behavioral database correctness is strictly
deferred to the differential harness.

\subsection{Stage 3: Differential Testing (\texttt{diff})}
The final stage is the \texttt{http-diff-job}, which performs a strict three-way
\ac{HTTP} comparison. It relies heavily on the executor's ability to pull sources
and artifacts, evaluating the compiled \texttt{HEAD} commit against two historical
baselines:
\begin{itemize}
	\item \textbf{Recent Baseline (\texttt{HEAD\textasciitilde 1}):}
		Identifies immediate regressions introduced by the developer's current
		push.

	\item \textbf{Fixed Baseline (\texttt{DIFF\_FIXED\_BASELINE}):} A pinned, stable
		anchor commit that tracks cumulative drift over the lifecycle of a long-running
		refactor.
\end{itemize}

To execute this without polluting the active workspace, the job leverages
\texttt{git worktree} to check out both baseline sources into isolated,
temporary directories. It compiles them cleanly from scratch to avoid cross-contamination
from stale artifacts. If a baseline is unavailable (for instance, if the
repository is new and lacks sufficient commit history), the harness
dynamically detects this and safely skips that specific sub-run.

\section{The HTTP-Level Diff}
\label{sec:httpdiff}

\texttt{HttpDiff.java} is the testing oracle. Because hand-writing expected outputs
for forty endpoints across five formats is unreliable, the harness uses
previous, trusted Git commits as free ground-truth oracles.

As detailed in Figure~\ref{fig:httpdiff-phases}, the harness boots up to
three Payara Micro subprocesses simultaneously on separate ports (\texttt{8081}
for last, \texttt{8082} for current, and \texttt{8083} for fixed).
Crucially, all instances are injected with the same \texttt{DATABASE\_URL}
environment variable, ensuring they query identical fixtures.

To make the testing strategy concrete, Listing~\ref{lst:diffcases} shows a representative
extract from the \texttt{http-diff-cases.json} test corpus. The file is structured
as an array of \ac{JSON} objects, where each object declaratively defines a request
profile.

\begin{lstlisting}[caption={Representative extract of the \texttt{http-diff-cases.json} test corpus.}, label={lst:diffcases}]
[
  { "name": "ping", "method": "GET", "path": "/api/ping" },
  { "name": "stations short csv", "method": "GET", "path": "/api/stations/station/short/csv", "timeoutSeconds": 120 },
  { "name": "station by id short json", "method": "GET", "path": "/api/stations/stationid/CASC00PRT/short/json" },
  { "name": "NEG unknown query param", "method": "GET", "path": "/api/stations/station/short/json?bogusparam=1" }
]
\end{lstlisting}

This declarative format allows the framework to evaluate roughly forty
specific scenarios across three distinct dimensions of the \ac{API}:
\begin{itemize}
	\item \textbf{Serialization and Volume Matrices:} Massive full-table
		queries (such as retrieving the entire station catalog) place high
		contention on the database when executed by three instances
		simultaneously. These cases are explicitly granted a \texttt{timeoutSeconds}
		boundary to prevent false failures while still guaranteeing byte-for-byte
		correctness across \ac{JSON}, \ac{XML}, \ac{CSV}, and GeoJSON formats.

	\item \textbf{Targeted Lookups:} Precise queries (e.g., filtering by the \texttt{CASC00PRT}
		marker or the \texttt{EUREF} network) verify that deeper \ac{SQL} joins and
		parameter bindings remain structurally intact after a refactor.

	\item \textbf{Negative Validation (\texttt{NEG}):} Cases prefixed with
		\texttt{NEG} intentionally inject malformed parameters, bogus output
		formats, or unregistered query keys. These ensure the "validate-then-query"
		architecture functions correctly, strictly intercepting bad requests with
		a \texttt{400 Bad Request} before they reach the persistence layer.
\end{itemize}

Every case in the corpus is then evaluated by the harness on a three-tier verdict
model:
\begin{itemize}
	\item \textbf{OK:} Both instances returned identical \ac{HTTP} statuses and
		normalized bodies.

	\item \textbf{WARN:} Both instances responded successfully, but their outputs
		differ (flagging an intentional or unintentional behavioral change).

	\item \textbf{FAIL:} One instance encountered an execution failure (e.g., a
		\texttt{5xx} server error, a connection refusal, or a timeout) while the
		other succeeded.
\end{itemize}

Because three parallel instances place high contention on a single database,
some heavy queries slow down. To prevent false failures, the harness
implements per-case timeout boundaries (\texttt{timeoutSeconds}), allowing
massive full-table list queries to process without globally lowering the
performance bar for faster endpoints.

\section{Reports and Findings}
\label{sec:reports}

Upon completion, the harness generates a self-contained report tree. The pipeline
uses \texttt{when: always} to guarantee these artifacts are uploaded even
if the job fails, as failing reports provide the most actionable diagnostic data.

\begin{itemize}
	\item \texttt{summary.json}: The entry point, containing pairwise verdict
		tallies and performance deltas (measuring if the current commit has become
		slower).

	\item \texttt{report.csv}: A row-by-row breakdown of all requests,
		including HTTP statuses and latency metrics.

	\item \texttt{details.txt}: A truncated, human-readable snippet of the
		differing payloads for non-OK requests, allowing developers to immediately
		spot the deviation.

	\item \texttt{payara-*.log}: The raw container logs for deep debugging.
\end{itemize}

Several operational realities shaped this reporting design. First, database
coupling is load-bearing: if the \texttt{FIXED\_SHA} anchor predates a
major schema change, it will query mismatched data, resulting in a wall of
bogus diffs. Thus, anchors must always represent compatible database eras. Second,
non-determinism does not equate to behavioral change. Early runs flagged \texttt{WARN}s
due to the unpredictable iteration order of Java \texttt{HashSet}s in error messages;
standardizing on sorted sets eliminated this cosmetic noise, yielding a highly
stable signal.
