\chapter{Conclusions and Future Work}
\label{chap:conc-trab-futuro}

\section{Main Conclusions}
\label{sec:conc-princ}

The work described in this report solves a concrete challenge for the \ac{SEGAL}
laboratory: safely refactoring a legacy Jakarta EE \ac{API} without
silently breaking existing \ac{EPOS} clients. The solution is a \ac{CI}/\ac{CD}-driven
differential testing framework that evaluates every new commit against two
baselines (the immediate parent and a fixed reference) by comparing raw \ac{HTTP}
responses.

Several key conclusions emerged from this project:
\begin{itemize}
	\item \textbf{Differential testing perfectly suits refactoring:} By
		comparing new outputs against known-good baselines, the framework avoids
		the brittleness of hard-coded expected values. It provided immediate, actionable
		feedback from the very first commit.

	\item \textbf{\ac{HTTP}-level comparison catches integration flaws:}
		Testing full \ac{HTTP} responses against a live database surfaces subtle
		regressions that isolated unit tests naturally miss. This is especially
		critical during a refactor, where database interactions are the most likely
		point of failure.

	\item \textbf{Self-hosted infrastructure is mandatory:} Because the tests
		require access to the production database, deploying a self-managed GitLab
		instance and runner within the laboratory's internal network was a strict
		data-locality requirement, not merely a preference.

	\item \textbf{Prototypes mitigate risk:} Building two isolated prototypes
		allowed for cheap experimentation with \ac{CI} configurations and Jakarta
		EE 11 features, keeping the production codebase clean of failed
		experiments.

	\item \textbf{Automation outperforms human review:} The framework
		successfully caught behavioral changes hidden inside seemingly cosmetic
		refactoring commits regressions that manual code review would likely
		have missed at the laboratory's standard commit rate.
\end{itemize}

\section{Future Work}
\label{sec:trab-futuro}

While the framework successfully meets its primary goals, several extensions
were deliberately left for future work to further mature the system:

\begin{itemize}
	\item \textbf{AI-Assisted Automated Triage:} Enhancing developer
		experience by integrating a lightweight \ac{LLM} analysis step at the end
		of the pipeline. By pushing the \texttt{summary.json} and \texttt{details.txt}
		artifacts to a serverless AI service (such as Cloudflare's AI \ac{REST}
		\ac{API}) via a simple \ac{HTTP} call, the system could automatically
		generate a structured, natural-language summary of any regressions.
		This would accelerate triage by posting immediate, human-readable feedback
		directly into the merge request, without requiring the laboratory to
		host its own AI infrastructure.

	\item \textbf{Endpoint URI Mapping (\texttt{prevURL}):} Currently, the harness
		assumes endpoint paths remain static. A necessary feature is the
		ability to map a \texttt{prevURL} to a new \ac{URI}. If an endpoint is renamed
		or versioned, the test should know to route the baseline request to the
		old URL and the candidate request to the new URL, ensuring the output remains
		identical.

	\item \textbf{Performance Benchmarking:} Implementing \textbf{BRIN} indexes
		on the \texttt{rinex\_file} date columns. This should be accompanied by a
		performance-regression subsystem that records wall-clock execution times
		against the large \texttt{DB\_3.0\_F} fixture to empirically measure
		speed improvements.

	\item \textbf{Spatial Database Migration:} Moving the custom spatial
		logic from the Java \texttt{Geometry} package directly into the database
		using \textbf{PostGIS}. The differential framework will be crucial here to
		guarantee byte-for-byte equivalence during the transition.

	\item \textbf{Automated Test Generation:} Dynamically generating the \ac{HTTP}
		test cases directly from the \ac{OpenAPI} document shipped inside the
		\ac{WAR}, ensuring newly added endpoints are automatically covered by
		the diff harness.
\end{itemize}

\bigskip

Overall, the framework delivered by this project solves the problem that
motivated it. It has already proven its value by catching real regressions during
the refactoring process, laying a solid foundation for the laboratory to
evolve the \ac{API} with a much higher degree of confidence. The future
work identified here represents natural next steps that build upon, rather than
invalidate, the current robust design.
