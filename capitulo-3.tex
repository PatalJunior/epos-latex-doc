\chapter{Technologies and Tools Used}
\label{chap:tecno-ferra}

\section{Introduction}
\label{chap3:sec:intro}

This chapter details the specific technologies and frameworks utilized to build
the solutions presented in this document. The selected stack is the result
of pragmatic engineering, balancing the established architecture of the production
\ac{API} with modern technical requirements. Rather than enumerating these tools
in isolation, the chapter examines the architecture layer by layer. It begins
with \ac{API} contracts and serialization formats, transitions through the Java
and Jakarta EE application ecosystem and its relational databases, and
finally covers the underlying servers and network tooling that support the
deployment.

\subsection{Explicit Data Serialization}
\label{sec:data-ser}

Unlike typical Jakarta \ac{REST} applications that rely on automatic
content negotiation via \texttt{Accept} headers and transparent message body
writers, the \ac{EPOS} \ac{API} handles serialization explicitly. Because endpoint
inputs consist solely of path and query parameters rather than incoming payload
bodies, the application bypasses automatic input deserialization entirely.

For outputs, the formatting logic is driven strictly by the \ac{URL} path
parameter. A dedicated parser component evaluates this parameter and routes the
data to the appropriate conversion method. Whether the requested format is
standard \ac{JSON}, its \ac{JSON-LD} or \ac{GeoJSON} variants, \ac{XML}, or \ac{CSV},
the parser explicitly converts the domain objects into a formatted text string.
The application then manually assigns the corresponding \ac{HTTP} \texttt{MediaType}
before constructing the final response.

This explicit architectural choice simplifies the validation strategy for
the differential testing harness. Because all serialization regardless of the
format ultimately produces a finalized string payload before leaving the Java
method, the test harness can treat the resulting \ac{HTTP} response body as an
opaque textual blob. It simply verifies the candidate's output byte-for-byte
against the parent commit's baseline, ensuring absolute formatting consistency
without needing format-specific parsing logic in the test suite.

\subsection{OpenAPI and Swagger UI}

The contract of the production \ac{API} is defined in an \ac{OpenAPI} 3.1.0~\cite{openapi}
document (\texttt{openapi.json}) packaged within the \ac{WAR}. This
document powers a Swagger \ac{UI}~\cite{swaggerui} instance hosted at \texttt{/swagger.html},
providing an interactive, in-browser environment to explore and test the
endpoints.

This setup yields two primary benefits. First, the document serves as the
definitive source of truth for downstream consumers of the \ac{API}, most
notably the \ac{EPOS} portal. Second, it enables future automated contract testing:
a planned evolution of the test framework, discussed in Chapter~\ref{chap:conc-trab-futuro},
involves dynamically deriving \ac{HTTP} differential test cases directly from
this \ac{OpenAPI} specification, rather than maintaining them as separate,
static \ac{JSON} definitions.

\section{Java and Jakarta EE}
\label{sec:java}

The applications developed in this project target Java SE 21~\cite{javaSE21}.
The decision to utilize Java was effectively dictated by the existing \ac{API}
codebase. However, modern language features are leveraged extensively: \texttt{record}
types are used liberally to model immutable request and response \acp{POJO},
while the new pattern-matching \texttt{switch} expressions replace several complex
\texttt{if/else} chains inherited from the legacy system, resulting in more maintainable
code.

\textbf{Jakarta EE 11}~\cite{jakartaee} serves as the foundational platform
layer, with the codebase fully migrated to the modern \texttt{jakarta.*}
namespace. The production application actively relies on a focused subset of
core specifications: Jakarta \ac{REST} 4.0 (\ac{JAX-RS})~\cite{jaxrs} is
used strictly for routing endpoints, while Jakarta \ac{CDI} 4.1 handles all
dependency injection (deliberately eschewing the standard JAX-RS \texttt{@Context}
injection). Furthermore, the application utilizes Jakarta Persistence 3.2 (\ac{JPA})~\cite{jpa}
for object-relational mapping and Jakarta Bean Validation 3.1 for
declarative constraints. Notably, standard Jakarta JSON Binding is omitted
in favour of explicit serialization via Jackson, as detailed in Section~\ref{sec:data-ser}.

During the early stages of the project, development prototypes served as a
learning sandbox to evaluate the broader capabilities of the Jakarta
ecosystem. These initial iterations experimented with Eclipse MicroProfile~\cite{microprofile}
features, specifically \ac{JWT} 2.1 for authentication mechanics and Health 4.0
for application probes. While these tools provided valuable insight into
the platform's offerings, they were ultimately excluded from the final application
in favour of a leaner, more tailored architecture.

Finally, alternative frameworks such as Spring Boot~\cite{springBoot} were briefly
considered but rejected. The existing production codebase is already built
upon Jakarta EE and deployed on an established Payara server infrastructure.
Attempting a parallel migration to a completely different ecosystem would have
introduced significant risk and overhead without providing any proportional technical
benefit.

\section{SQL and the Database Layer}
\label{sec:sql}

The persistence model of the production \ac{API} is heavily relational: \ac{GNSS}
stations, networks, agencies, countries, file types, observation summaries, and
embargo windows are all inextricably linked through foreign keys. Because
users require highly specific data filtering, most endpoints must execute complex,
conditional joins per request.

While \ac{JPA} is utilized for its foundational object-relational mapping,
the sheer complexity of the API's query requirements makes purely object-oriented
querying (such as the \ac{JPA} Criteria API) unwieldy. Instead, developers actively
construct dynamic, hand-made queries through conditional string
concatenations directly in Java. These hand-crafted strings are then passed to
the \ac{JPA} \texttt{EntityManager}, which safely binds the parameters and
maps the resulting data rows back into Java \acp{POJO}. Consequently, native
\ac{SQL} and its structural logic remain the true driving force of the
database layer.

The laboratory has standardized on \textbf{PostgreSQL}~\cite{postgresql} for
the \ac{EPOS} production database. The decision to use PostgreSQL over alternatives
like \textbf{MySQL}~\cite{mysql} was driven primarily by historical
precedent within the lab, as well as its robust support for specialized index
types (such as B-tree, BRIN, and GiST) that are highly effective for large scientific
datasets. Furthermore, while the PostGIS extension is not actively utilized today,
standardizing on PostgreSQL provides a realistic, low-friction pathway for future
spatial data integrations.

\section{Servers, Tooling and Network}
\label{sec:servers}

\textbf{Payara Server}~\cite{payara} is the Jakarta EE application server adopted
by the laboratory; the production \ac{API} is simply deployed onto an existing
Payara installation. Its companion product, \textbf{Payara Micro}, is a single,
self-contained \ac{JAR} that launches a deployable \ac{WAR} directly from
the command line. Because Payara was already the established stack,
adopting Payara Micro for test orchestration was the immediate and obvious
choice. It serves as the engine for the \ac{HTTP}-level differential
harness, booting one instance per \ac{WAR} on different ports to safely
replay and compare requests.

The applications are built using \textbf{Apache Maven}. The project's
\texttt{pom.xml} orchestrates the build lifecycle, utilizing the \texttt{maven-war-plugin}
to produce the deployable \texttt{ROOT.war}, the \texttt{maven-failsafe-plugin}
for integration testing (outputting JUnit \ac{XML} reports), and the
\texttt{maven-dependency-plugin} to stage the Payara Micro \ac{JAR} so the
differential harness can spawn it as a subprocess.

\textbf{GitLab}~\cite{gitlabci} handles both source control and \ac{CI}/\ac{CD}
orchestration. The laboratory hosts a self-managed GitLab instance~\cite{gitlabSelfManaged},
while the runner operates on a separate internal \ac{VM} executing jobs
inside \textbf{Docker}~\cite{docker} containers. Pipeline jobs utilize the
official \texttt{maven:\allowbreak 3.9.6-eclipse-\allowbreak temurin-21} image
to guarantee a clean, reproducible toolchain. Under the hood, \textbf{Git}~\cite{git}
is used heavily in the pipeline: the \texttt{git worktree} command materializes
the baseline build alongside the candidate build on the runner's filesystem without
disrupting the working copy (detailed in Section~\ref{sec:pipeline}).

During early development, local database instances were simulated using
Docker containers. However, this was later abandoned in favor of connecting
directly to the real development database. Because the GitLab runners are provisioned
on the same internal network as the \ac{EPOS} database, connecting to the actual
infrastructure was the most obvious and reliable choice for integration testing.
The production \ac{API} itself remains uncontainerized, deployed
traditionally by copying the \ac{WAR} onto the existing Payara server.

Finally, the production database is strictly isolated within the laboratory's
internal network. While the internal runners can access it directly,
developers operating remotely must connect via the university's established
\textbf{OpenVPN} infrastructure, managed through pfSense. This setup utilizes
existing \ac{UBI} certificates, relying on proven, pre-configured network
infrastructure rather than introducing new \ac{VPN} solutions to the laboratory.

\section{Conclusion}
\label{chap3:sec:concs}

The technology stack on which this project is built is in large part
imposed by its environment: Jakarta EE on Payara, PostgreSQL on the
internal network, GitLab as the self-managed \ac{CI} platform. Where there was
room for choice, between Jakarta EE and Spring Boot, between Payara Server and
Apache Tomcat, between PostgreSQL and MySQL, between JUnit 5 and TestNG,
the decisions documented in this chapter were guided by a preference for
fewer moving parts, by the alignment with what the laboratory already runs in
production and by the desire to keep the differential testing harness as simple
as possible. The next chapter describes the iterative methodology by which
these technologies were combined into a working solution.
